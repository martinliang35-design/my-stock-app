# 项目稳定性：根因分析与增强方案

针对「文件丢失」与「重启后 404/500」两类不稳定问题做根因归纳，并通过版本控制规范、统一重启脚本与文档完善来增强项目稳定性。本文档为方案归档，具体已落实项见各相关文档与 `package.json` 中的 `dev:restart` 脚本。

---

## 一、根因归纳

### 1. 文件丢失（多文件夹为空/文件没了）

| 维度 | 说明 |
|------|------|
| **直接原因** | 业务代码（`docs/`、`lib/`、`components/`、`app/api/`、`supabase/migrations/` 等）从未被 Git 跟踪（untracked）。 |
| **触发条件** | 执行了会删除未跟踪文件或还原/覆盖项目的操作，例如：`git clean -fd`、`git reset --hard` 后误清、IDE「清理未跟踪文件」、或用「仅含初始提交」的副本覆盖了当前目录。 |
| **已有文档** | [ROOT_CAUSE_文件丢失分析.md](./ROOT_CAUSE_文件丢失分析.md) 已记录完整根因与操作对照。 |

**结论**：根因是「业务代码长期未提交 + 某次操作清理或覆盖了工作区」，不是 Git 损坏或单次误点，而是版本控制习惯与操作风险叠加。

---

### 2. 重启后 404 / 500（重启后打不开）

| 现象 | 根因 | 当前状态 |
|------|------|----------|
| **404** | 开发缓存与进程不一致：杀进程或异常退出 `next dev` 后，`.next/dev`、`.next/cache` 处于脏状态，Turbopack 路由解析用了过期缓存，根路径 `/` 被当成未找到。 | 已有排查步骤（清缓存、删锁、再启），见 [开发环境问题排查.md](./开发环境问题排查.md)。 |
| **500** | `lib/supabase.ts` 在**模块加载时**缺 env 就 `throw`。服务端渲染会加载该模块，重启后若 env 未注入或未就绪，服务端抛错 → Internal Server Error。 | **已修复**：改为仅在客户端校验并抛错，服务端用占位值创建 client，避免 SSR 时 500。 |

**结论**：404 来自 Next.js 开发缓存/多实例/端口；500 来自服务端加载 Supabase 时的环境变量校验。两者都与「重启后的环境与缓存状态」有关。

## 二、稳定性增强方案（已实施摘要）

### 2.1 防止文件再次丢失（版本控制与习惯）

- **一次性补救（若尚未做）**  
  将当前所有业务代码纳入一次提交，便于以后用 Git 恢复：
  ```bash
  git add docs/ lib/ components/ app/ supabase/ README.md package.json package-lock.json
  git commit -m "chore: 提交完整业务代码与文档，纳入版本控制"
  ```
  若存在未跟踪的配置文件（如 `next.config.ts`、`tsconfig.json` 等有改动），一并 `git add` 后提交。

- **长期习惯**  
  - 每完成一个功能或修复就 `git add` + `git commit`，避免大块未跟踪/未提交代码。  
  - 慎用 IDE/命令行中的「清理未跟踪文件」「Discard all」；使用前确认无重要未提交目录。

- **可选：远程备份**  
  `git remote add origin <url>` 后 `git push -u origin main`，本机误操作仍可从远程拉回。

- **文档**  
  [ROOT_CAUSE_文件丢失分析.md](./ROOT_CAUSE_文件丢失分析.md) 第 6 节「稳定性：首次克隆/恢复后必做」；[README.md](./README.md) 文档列表中已索引该文。

---

### 2.2 开发服务器稳定启动（统一重启流程）

- **已实施**  
  - `package.json` 中新增脚本 `dev:restart`：结束 3000/3001 端口进程 + 清理 `.next/dev`、`.next/cache` + 启动 `next dev`。  
  - [开发环境问题排查.md](./开发环境问题排查.md)：一、404 增加「推荐做法」使用 `npm run dev:restart`；新增「五、重启后 500」说明根因与已修复。

- **日常建议**  
  重启开发服务器时优先执行 `npm run dev:restart`，减少 404 或起不来。

---

### 2.3 环境变量模板（已实施）

- 项目根目录已提供 `.env.example`，列出 `NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY`。  
- 根目录 [README.md](../README.md) 与 [technical.md](./technical.md) 中说明：首次克隆可复制为 `.env.local` 并填入真实值。

## 四、小结

- **文件丢失**：根因是业务代码未纳入 Git + 清理/还原/覆盖类操作；通过「一次性提交 + 提交习惯 + 文档」加固。  
- **重启 404/500**：404 因 Next 开发缓存/多实例；500 因服务端加载 Supabase 时抛错（已修复）。通过「dev:restart 脚本 + 文档」统一重启流程并记录 500 根因。

按上述执行后，项目在「不丢文件」与「重启可预期」两方面会明显更稳定，且不依赖额外基础设施，仅用 Git、npm 脚本与文档即可完成。
