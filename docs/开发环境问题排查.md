# 开发环境问题排查

## 一、访问 localhost:3000 出现 404（This page could not be found）

**推荐做法**：日常重启开发服务器时，优先执行 `npm run dev:restart`（会先结束占用 3000/3001 的进程、清理 `.next/dev` 与 `.next/cache`，再启动 dev），可减少因缓存或端口冲突导致的 404。

### 根因（按可能性）

1. **开发缓存与进程不一致**  
   杀过占用 3000 端口的进程、或异常退出过 `next dev` 后，`.next/dev` 和 `.next/cache` 可能处于脏状态，Turbopack/Next 在解析路由时用了错误或过期的缓存，导致根路径 `/` 被当成“未找到”。

2. **多实例冲突**  
   同时运行了多个 `next dev`（例如一个在终端、一个在 IDE），后启动的拿不到锁或用了错误端口，你访问的端口对应的实例没有正确编译首页。

3. **端口搞错**  
   若终端里提示 “Port 3000 is in use, using 3001 instead”，实际服务在 3001，访问 3000 可能得到别的进程或空响应，有时会表现为 404。

### 处理步骤（建议按顺序做）

1. **结束占用端口的进程**
   ```bash
   lsof -ti :3000 | xargs kill -9
   lsof -ti :3001 | xargs kill -9
   ```

2. **删开发缓存（关键）**
   ```bash
   rm -rf .next/dev .next/cache
   ```

3. **删除锁文件（若曾报 Unable to acquire lock）**
   ```bash
   rm -f .next/dev/lock
   ```

4. **重新启动开发服务器**
   ```bash
   npm run dev
   ```

5. **只访问终端里提示的地址**  
   例如 `http://127.0.0.1:3000`，若显示 3001 就访问 3001。

若仍 404，可再执行一次完整清理后重启：
```bash
rm -rf .next && npm run dev
```

## 二、页面一直“加载中…”

- **根因**：`AuthGuard` 里调用的 `supabase.auth.getSession()` 一直不返回（网络慢、Supabase 不可达等）。
- **已做**：10 秒超时后自动进入登录页，并提示“首次加载超时，请检查网络后重试登录。”
- **你可做**：检查网络/代理，或稍后重试；登录/注册请求也有 25 秒超时与中文错误提示。

---

## 三、ERR_NETWORK_IO_SUSPENDED / 连接已中断

- **根因**：电脑进入休眠后，系统挂起网络 I/O，浏览器与 localhost 断开。
- **处理**：唤醒电脑 → 在项目目录重新执行 `npm run dev` → 浏览器点“重新加载”或重新打开 `http://localhost:3000`。

---

## 四、Unable to acquire lock at .next/dev/lock

- **根因**：已有别的 `next dev` 在跑，或上次异常退出留下锁文件。
- **处理**：先结束占用 3000/3001 的进程（见“一”），再删锁 `rm -f .next/dev/lock`，然后只启动一次 `npm run dev`。或直接使用 `npm run dev:restart`。

---

## 五、重启后 500（Internal Server Error）

- **根因**：此前 `lib/supabase.ts` 在模块加载时若缺少环境变量会直接 `throw`，服务端渲染会加载该模块，导致 SSR 抛错并返回 500。已改为**仅在客户端**校验并抛错，服务端用占位值创建 client，避免重启后因 env 未就绪而 500。
- **若仍出现 500**：请确认项目根目录存在 `.env.local`，且已配置 `NEXT_PUBLIC_SUPABASE_URL` 与 `NEXT_PUBLIC_SUPABASE_ANON_KEY`；重启后使用终端里实际提示的地址（如 `http://127.0.0.1:3000` 或 3001）访问。
